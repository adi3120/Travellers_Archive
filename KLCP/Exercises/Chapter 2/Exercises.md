- This is wrong, each shell maintains its own job table and cant know about other shell's job table
- Jobs are processes generated by a terminal (shell) while processes are any programs running on a computer.
-  In other words, jobs are defined from a user point of view while processes are defined from an operating system point of view.
- Jobs are separately managed by each terminal.
- Job ID
	- Each job generated by a terminal has a unique ID called **job ID**. It is unique within the same terminal but the same ID number can be used for other terminals.
- Process ID
	- Each process also has a unique ID called **process ID**
	- Each process is unique within the same operating system (unique even among different terminals connecting to the same operating system)
	- When a job is created, a process ID is assigned to the job
	- you can manage the job using either job ID or process ID
	- Process IDs are also assigned to system-generated processes which don't have job IDs.

# find vs locate

`find` and `locate` both help you find files, but they work very differently.

|Feature|`find`|`locate`|
|---|---|---|
|Searches|Filesystem directly|Prebuilt database|
|Speed|Slower|Very fast|
|Accuracy|Always current|Can be outdated|
|Search by|Name, size, permissions, time, owner, etc.|Mostly pathname|
|Requires indexing|No|Yes (`updatedb`)|

---

## `find`

`find` walks through directories and examines files in real time.

Example:

```bash
find /home -name "notes.txt"
```

Searches the filesystem right now.

You can also search by many attributes:

```bash
find . -size +10M
```

Files larger than 10 MB.

```bash
find . -type f -mtime -7
```

Files modified within the last 7 days.

```bash
find . -perm 755
```

Files with permission 755.

### Pros

- Always up-to-date
    
- Extremely flexible
    

### Cons

- Can be slow on large filesystems
    

---

## `locate`

`locate` searches a database of filenames.

Example:

```bash
locate notes.txt
```

Returns results almost instantly because it isn't scanning the disk.

### How it works

The database is built by:

```bash
sudo updatedb
```

Then:

```bash
locate notes.txt
```

searches that database.

### Pros

- Very fast
    
- Easy to use
    

### Cons

- Database may be stale
    
- Cannot search by size, permissions, modification time, etc.
    

---

## Example

Suppose you create a file:

```bash
touch secret.txt
```

Immediately:

```bash
find . -name secret.txt
```

finds it.

But:

```bash
locate secret.txt
```

may **not** find it until the database is updated:

```bash
sudo updatedb
```

---

## When to use which?

Use **`locate`** when:

- You roughly know the filename
    
- You want results quickly
    

```bash
locate firefox
```

Use **`find`** when:

- You need accurate, current results
    
- You need advanced filters
    

```bash
find /var/log -name "*.log" -mtime -1
```

A common workflow is:

```bash
locate filename
```

to quickly find likely locations, then

```bash
find /path -name filename
```

for precise searching.

## 2.6.1 Exploring Devices

```bash
file /dev/snd/seq
```

Output:

```text
/dev/snd/seq: character special (116/1)
```

```bash
file /dev/sda1
```

Output:

```text
/dev/sda1: block special (8/1)
```

- Sound device = Character device
    
- Disk partition = Block device

## 2.6.3 Searching Files

Display kernel messages:

```bash
sudo dmesg | more
```

Find file:

```bash
find / -name rockyou.txt.gz
```

Locate file:

```bash
locate rockyou.txt.gz
```

Measure execution time:

```bash
time find / -name rockyou.txt.gz
```

```bash
time locate rockyou.txt.gz
```

### Why is `locate` faster?

`locate` searches a pre-built database instead of scanning the entire filesystem like `find`.
